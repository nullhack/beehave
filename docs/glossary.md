# Glossary: beehave

A living glossary of terms used across the beehave project. Updated as the
product vocabulary evolves.

---

## B

### Backlog stage
`docs/features/backlog/` — organizational subfolder for `.feature` files that
represent work not yet started. Stubs generated from backlog features map to
the same `tests/features/` path as features in any other stage.

### Beehave
The framework-agnostic Python library that keeps Gherkin `.feature` files and
test stubs synchronized. Consumed as a CLI (`beehave <command>`) and as a
Python package (`from beehave import ...`).

### Body (test)
The implementation code inside a test function — the part between the signature
and the closing line. Beehave **never** modifies test bodies.

## C

### Cache
`.beehave_cache/features.json` — advisory hash cache that lets beehave skip
unchanged `.feature` files during sync. Auto-rebuilds silently if stale, missing,
or corrupted. Ignored by git.

### Completed stage
`docs/features/completed/` — organizational subfolder for `.feature` files that
represent shipped or accepted work. No behavioral difference from backlog or
in-progress.

### Config
The `[tool.beehave]` section in `pyproject.toml`. Single source of truth for
runtime settings. CLI flags override config values.

## D

### Deprecation marker
Framework-specific tag applied to a test stub when the corresponding Gherkin
Example (or its parent Rule or Feature) carries a `@deprecated` tag. For pytest:
`@pytest.mark.deprecated`.

### Docstring (test)
The triple-quoted string inside a test function. Beehave re-renders this to
match the current Gherkin scenario text verbatim during `sync-update`.

## E

### Example
A Gherkin scenario (or Scenario Outline instance) inside a `.feature` file.
Every Example must have an `@id` tag for stub mapping to work.

## F

### Feature file
A `.feature` file containing Gherkin `Feature:`, `Rule:`, and `Example:` blocks.
The source of truth for beehave. Beehave writes to `.feature` files **only** to
add missing `@id` tags.

### Feature slug
The kebab-case stem of a `.feature` file converted to snake_case, lowercase.
`user-authentication.feature` → `user_authentication`. Used in test directory and
function naming.

### Framework adapter
A pluggable module that translates Examples into framework-specific test stubs.
For v1 only the pytest adapter is built-in. Adapters provide skip, deprecated,
parametrize, and header templates.

## H

### Hatch
`beehave hatch` — the CLI command that generates bee-themed demo `.feature`
files for onboarding. Pure content generation; does not create directories.

## I

### ID
`@id:<8-char-hex>` — stable identifier attached to every Gherkin Example.
Generated by beehave when missing. Developer-supplied IDs are respected as-is.
Uniqueness is project-wide across all `.feature` files.

### In-progress stage
`docs/features/in-progress/` — organizational subfolder for `.feature` files that
represent work currently being built. No behavioral difference from backlog or
completed.

## M

### Malformed ID
An `@id` tag with an empty value (`@id:`) or non-hex characters
(`@id:ZZZZZZZZ`). Treated as missing; beehave generates a replacement.

## N

### Nest
`beehave nest` — the CLI command that bootstraps the canonical directory
structure (`docs/features/{backlog,in-progress,completed}/`, `tests/features/`).
Additive and idempotent.

## O

### Orphan stub
A test function whose `@id` no longer matches any Example in the project.
Beehave warns only — it never deletes or modifies orphan stubs.

## P

### Parametrize template
The adapter-supplied template that renders framework-native parameter
declarations for Scenario Outlines (e.g. `@pytest.mark.parametrize(...)` for
pytest).

### Pytest adapter
The default framework adapter for v1. Generates flat test functions with
`test_` prefix, `@pytest.mark.skip`, `@pytest.mark.deprecated`,
`@pytest.mark.parametrize`, `-> None`, and `...` body.

## R

### Rule
A `Rule:` block inside a `.feature` file. Maps to one test file:
`tests/features/<feature_snake>/<rule_slug>_test.py`.

### Rule slug
The slugified title of a Gherkin `Rule:` block, lowercase with underscores.
Used in test file naming.

## S

### Skip marker
Framework-specific tag applied to every newly generated stub so the test is
skipped until the developer implements the body. For pytest:
`@pytest.mark.skip(reason="not yet implemented")`.

### Status
`beehave status` — dry-run preview of what `sync` would do. Exits 0 if in sync,
1 if changes pending. Silent by default; `--verbose` and `--json` for detail.

### Stub
A test function generated by beehave for an Example. Contains auto-generated
parts (name, markers, docstring) and a developer-owned body.

### Sync
`beehave sync` — the main command that scans `.feature` files and reconciles
them with test stubs (create, update, warn). One framework per invocation.

## T

### Template customization
User-defined template folder that overrides built-in adapter templates. Specified
via `--template-dir` or `template_path` in `[tool.beehave]`. Only files present
in the custom folder override; missing files fall back to built-in.

### Test directory
`tests/features/<feature_snake>/` — the directory where a feature's test files
live. Mapped identically regardless of which stage subfolder the `.feature` file
resides in.

### Test file
`<rule_slug>_test.py` — one per Rule inside a feature's test directory.

## U

### Unittest adapter
PARKED for v2. Will generate `unittest.TestCase` method stubs. Not part of v1.
